# HashPay 协作备忘

本文件是仓库内唯一协作约束源。讨论确认后只改这里，避免多份文档冲突。

## 编码风格硬约束

1. 内部流转数据默认可信，不做过度防御性编程，不无故增加复杂度。
2. 性能优先，函数级别整体理解后再下手，追求简洁与可读。
3. 不做不必要的常量、函数、定义；低变更概率值可在函数内联。
4. 架构保持清晰稳定，便于双方理解和长期维护。
5. 在可追踪调用链的前提下，不加无意义的空指针检查。
6. 优先复用现有逻辑，避免重复实现导致维护成本上升。
7. 命名清晰表意即可，变量名/函数名一般控制在 2-3 个单词。
8. 只在关键部分写注释，帮助理解，不堆砌注释。
9. `go test` 不是唯一目标，优先覆盖跨模块关键流程可用性。

## 输出与文案规范

1. 区分“用户提示”和“开发调试信息”。
2. 用户提示只写可执行结论与下一步动作，不暴露无意义内部细节（例如内部探测路径）。
3. 技术细节（底层错误、内部请求地址、堆栈线索）放 `DEBUG` 日志。
4. 报错文案优先可理解、可操作，例如“检查域名解析/证书/反向代理”。

## 变更验证策略

1. 仅文案或注释改动：执行 `gofmt`，默认不跑测试。
2. 局部逻辑改动：只跑对应专项测试。
3. 核心流程或跨模块改动：跑全量测试。

## 项目要求

1. 以 `miniapp` 为核心，偏 UI/管理工具化体验。
2. 允许根据需求重构项目结构、功能、命名，不被早期方案绑定。
3. 方向先对齐，再实现；实现过程可迭代调整。

## 项目目标

1. 以 Telegram Mini App 作为管理后台，绝大部分配置在后台完成。
2. 项目本质是加密支付网关；不是 Telegram 生态工具本身。
3. 除支付网关核心能力外，绝大部分交互表现放在 Telegram（Bot/Mini App）。
4. 项目形态为前后端分离。
