# HashPay 协作备忘

本文件是仓库内唯一协作约束源。讨论确认后只改这里，避免多份文档冲突。

## 项目定位

1. HashPay 的本质是“加密支付网关”，不是 Telegram 工具项目。
2. Telegram Bot 和 Telegram Mini App 是管理入口与交互载体，不是产品目标本身。
3. 项目形态是前后端分离：
   后端负责订单、支付路由、扫描、通知、Bot 协调。
   `miniapp` 负责管理员工作台。
   `web` 负责付款用户的托管收银台。
4. 当前阶段允许破坏性重构，不保留兼容性，不为旧接口、旧字段、旧页面额外加兼容层。

## 当前系统意图

1. 管理员的核心任务不是“看很多页面”，而是尽快完成支付配置、查看待处理订单、确认支付运行状态、管理商户接入。
2. 付款用户的核心任务不是“理解系统”，而是在最短路径内完成付款，所以收银台必须围绕订单摘要、倒计时、支付方式选择、付款指引展开。
3. 系统优先提供托管收银台能力：
   商户创建订单。
   用户打开支付页。
   用户选择支付方案。
   系统生成支付快照。
   扫描到账。
   系统回调商户。
4. Inline Bot 收款是订单入口的一种，不是独立支付体系。

## 代码风格硬约束

1. 内部流转数据默认可信，不做过度防御性编程，不无故增加复杂度。
2. 性能优先，函数级别整体理解后再下手，追求简洁与可读。
3. 不做不必要的常量、函数、定义；低变更概率值可在函数内联。
4. 架构保持清晰稳定，便于双方理解和长期维护。
5. 在可追踪调用链的前提下，不加无意义的空指针检查。
6. 优先复用已有正确逻辑，避免重复实现。
7. 命名清晰表意即可，变量名/函数名一般控制在 2-3 个单词。
8. 只在关键部分写注释，帮助理解，不堆砌注释。
9. `go test` 不是唯一目标，优先覆盖跨模块关键流程可用性。
10. 相比“打补丁”，优先做最小但正确的完整方案。

## 后端架构约束

当前后端按职责分层，后续改动不要再把职责揉回旧式大杂烩：

1. `internal/app`
   负责配置模型、首次启动引导、依赖装配、运行时启停。
2. `internal/http`
   只负责 Web 服务、路由、中间件、HTTP DTO、静态资源挂载。
3. `internal/bot`
   只负责 Telegram 协议交互、消息场景分发、inline 订单交互。
4. `internal/models`
   负责数据库、模型、按功能拆分的数据访问与迁移。
5. `internal/payments`
   负责支付驱动定义、驱动注册、表单 schema、报价、支付分配、扫描扩展。
6. `internal/service`
   负责领域用例与跨模块业务编排。
7. `internal/jobs`
   负责过期、扫描、通知等后台任务。
8. `internal/utils`
   只保留真正通用的工具代码。

禁止继续把业务逻辑横散到 handler、bot、scanner 里彼此直接拼接。

## 领域模型约束

当前统一围绕以下实体建模：

1. `Merchant`
   商户主体，包含名称、API Key、签名密钥、默认回调、状态。
2. `Order`
   订单主体，包含商户单号、计价金额、来源、状态、回调状态。
3. `PaymentMethod`
   管理员配置的支付方式实例，归属某个支付驱动。
4. `PaymentRoute`
   订单选择后的最终支付快照，包含驱动、网络、币种、地址/账户、金额、二维码值、说明。
5. `PaymentTx`
   扫描确认后的交易记录。
6. `NotificationTask`
   商户回调任务。
7. `AdminUser`
   Telegram 管理员主体。

订单主状态收敛为：

1. `pending`
2. `paid`
3. `expired`
4. `invalid`

不要再给主流程新增含混状态，例如旧的 `failed`。

## 支付模块约束

1. 支付方式必须模块化，围绕驱动组织，不允许把链、币、地址、API Key 直接散在 handler 或页面逻辑里。
2. 驱动至少表达：
   元信息。
   管理表单 schema。
   报价能力。
   支付分配能力。
   扫描适配能力。
3. 当前 v1 范围是：
   链上支付。
   交易所内部转账。
4. 第三方钱包先不做独立驱动；后续若接入，也应走同一驱动模型。
5. 订单页、Bot、扫描器都应围绕 `PaymentRoute` 工作，不再反查 payment method 临时拼信息。

## Bot 场景约束

Bot 明确分四种场景，不允许再用一个文本入口兜底所有情况：

1. `setup`
   机器人未初始化时，仅处理验证码验证与管理员绑定。
2. `admin`
   已初始化后，管理员命令、状态查看、Mini App 入口、inline 建单。
3. `public`
   已初始化后，普通用户触达机器人时的支付说明与引导。
4. `inline_checkout`
   通过 inline 创建收款订单后的消息生命周期。

Banner 规则固定：

1. 只有 `inline_checkout` 使用 banner 占位。
2. 一旦订单消息进入二维码图片态，只更新 caption / 按钮，不再尝试恢复成纯文本消息。
3. 普通管理消息、帮助消息、说明消息默认不用 banner。

## 管理后台约束

1. `miniapp` 是管理员工作台，不是 BI 展示墙。
2. 页面优先级固定为：
   Dashboard。
   Orders。
   Payments。
   Merchants。
   Settings。
3. Dashboard 优先展示待处理事项、今日关键指标、支付健康状态、最近订单。
4. Payment 页面必须以 schema 驱动表单为主，不再为每个支付方式手写独立表单。
5. Merchants 页面聚焦接入管理，不扩展无关 CRM 功能。
6. 保留 Telegram iOS / Android 差异适配，但统一收口到通用组件层，不散在页面细节里。

## 收银台约束

1. `web` 只围绕托管收银台，不承担管理后台职责。
2. 收银台必须围绕以下主链路组织：
   商户信息与订单摘要。
   倒计时。
   币种选择。
   支付方案选择。
   当前支付卡片。
   已支付 / 已过期结果态。
   帮助与订单号复制。
3. 首屏要优先帮助用户完成付款，不做后台式信息堆砌。
4. 交易所支付方案要展示账户、备注、步骤说明，不强行套二维码。
5. 过期订单不允许继续选择支付方案。

## API 约束

公开接口按以下边界组织：

1. `Setup API`
   初始化与运行状态。
2. `Admin API`
   后台管理能力。
3. `Merchant API`
   商户下单与查单。
4. `Checkout API`
   付款页读取与支付方案选择。

约束如下：

1. 管理端接口返回页面友好数据，不把拼装压力丢给前端。
2. 商户接口只保留托管网关模型，不继续扩展旧式散乱接口。
3. 收银台只围绕 `checkout` 资源，不直接读取后台支付配置结构。

## 输出与文案规范

1. 区分“用户提示”和“开发调试信息”。
2. 用户提示只写可执行结论与下一步动作，不暴露无意义内部细节。
3. 技术细节（底层错误、内部请求地址、堆栈线索）放 `DEBUG` 日志。
4. 报错文案优先可理解、可操作，例如“检查域名解析/证书/反向代理”。

## 变更验证策略

1. 仅文案或注释改动：执行 `gofmt`，默认不跑测试。
2. 局部逻辑改动：只跑对应专项测试。
3. 核心流程或跨模块改动：跑全量测试。
4. 前端页面或接口改动后，除构建通过外，还要确认主链路能闭合：
   初始化。
   管理配置。
   商户下单。
   收银台选路由。
   扫描确认。
   商户回调。

## 协作原则

1. 先从原始需求出发，不默认需求已经被完全想清楚。
2. 只有在关键歧义会导致明显不同方案时才先澄清；否则基于最合理解释推进，并明确假设。
3. 输出方案前，按输入、处理流程、状态变化、输出、上下游影响做链路检查。
4. 不擅自扩展业务目标，不引入与当前需求无关的兜底和旁支设计。
5. 可以重构结构、命名、模块边界，不被旧实现绑定，但要维持新的整体一致性。
